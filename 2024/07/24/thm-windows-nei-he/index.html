THM-Windows内核

hybcx2024-07-242024-08-01

学习并了解 Windows 核心运行方式的基础知识。

0x01 介绍

操作系统背后的技术和架构比我们最初看到的要多得多。在这个房间里，我们将观察Windows操作系统和常见的内部组件。

学习目标

  • 了解 Windows 进程及其底层技术并与之交互。
  • 了解核心文件格式及其使用方式。
  • 与 Windows 内部进行交互并了解 Windows 内核的运行方式。
    由于 Windows 机器构成了企业基础设施的大部分，红队需要了解 Windows 的内部结构以及如何（滥用）使用它们。在制作攻击性工具或漏洞利用时，红队可以（滥用）使用 Windows 来帮助规避和利用。

在开始本课程之前，请先熟悉 Windows 的基本用法和功能。还建议您具备 C++ 和 PowerShell 的基本编程知识，但不是必需的。

我们提供了一台基本的 Windows 机器以及完成这个房间所需的文件。您可以使用以下凭据在浏览器中或通过 RDP 访问计算机。

Machine IP: MACHINE_IP             Username: THM-Attacker             Password: Tryhackme!

这将包含大量信息。请系好安全带并找到最近的灭火器。

0x02 Process

进程维护并代表程序的执行；一个应用程序可以包含一个或多个进程。一个进程有许多组件，它被分解成要存储和交互的组件。微软文档对这些其他组件进行了分解，“每个进程都提供执行程序所需的资源。进程具有虚拟地址空间、可执行代码、系统对象的开放句柄、安全上下文、唯一的进程标识符、环境变量、优先级、最小和最大工作集大小以及至少一个执行线程。”这些信息可能看起来令人生畏，但这个房间的目的是让这个概念变得不那么复杂。

如前所述，进程是通过应用程序的执行而创建的。进程是 Windows 运行的核心，Windows 的大多数功能都可以包含为应用程序并具有相应的进程。以下是启动进程的默认应用程序的一些示例。

  • MsMpEng (Microsoft Defender)
  • wininit (keyboard and mouse)
  • lsass (credential storage)

攻击者可以瞄准进程来逃避检测并将恶意软件隐藏为合法进程。以下是攻击者可以针对进程使用的一小部分潜在攻击向量，

  • Process Injection (T1055) – 进程注入
  • Process Hollowing (T1055.012) – 进程镂空
  • Process Masquerading (T1055.013) – 进程伪装

0x03 Threads

线程是进程使用的可执行单元，并根据设备因素进行调度。

设备因素可能因 CPU 和内存规格、优先级和逻辑因素等而异。

我们可以简化线程的定义：“控制进程的执行”。

由于线程控制执行，因此这是一个常见的目标组件。线程滥用可以被单独使用来帮助代码执行，或者更广泛地用于与其他 API 调用链接，作为其他技术的一部分。

线程与其父进程共享相同的细节和资源，例如代码、全局变量等。线程也有其独特的值和数据，如下表所示。

0x04 虚拟内存

虚拟内存是Windows内部机制工作以及相互作用的一个关键组件。虚拟内存允许其他Windows内部组件与内存进行交互，就像它是物理内存一样，而不会造成应用程序之间发生冲突。

虚拟内存会为每个进程提供一个私有的虚拟地址空间，而内存管理器则用于将虚拟地址转换为物理地址。通过使用私有虚拟地址空间而不直接写入物理内存，可以降低进程造成损害的风险。

内存管理器还将使用分页或传输来处理内存。应用程序所使用的虚拟内存可能会多于被分配的物理内存，为了解决这个问题，内存管理器将通过传输或分页的方式，暂时将虚拟内存交换到磁盘上，我们可以通过下面的图像可视化地了解这个概念。

在32位x86系统上，理论上的最大虚拟地址空间为4GB。这个地址空间可以被分为两半，下半部分(0x00000000 - 0x7FFFFFFF)将被分配给如前所述的进程，而上半部分(0x80000000 - 0xFFFFFFFF)则会被分配给操作系统内存利用，管理员可以通过设置(increaseUserVA)或者AWE(Address Windowing Extensions)来为需要更大地址空间的应用程序更改此分配布局。

tips：AWE-地址窗口扩展，是Windows应用程序接口，允许32位应用程序访问超出其虚拟地址空间限制的物理内存，AWE将虚拟地址空间映射到物理内存的过程被称为“加窗”，这对于数据密集型应用程序有益。

在64位现代系统上，理论上的最大虚拟地址空间为256TB，并且64位系统沿用了32位系统的确切地址布局比例。

"确切地址布局比例"可以理解为地址空间与地址总线位宽之间的一种对应关系和比例，这种精确的地址布局方式从 32 位系统中被完整继承到 64 位系统中。

我们知道64位系统的最大虚拟地址空间扩大到了256TB，那么如何理解地址空间与地址总线位宽的比例(确切地址布局比例)仍然保持一致呢?
主要有以下几点:
1. 64位系统扩展了地址总线位宽到64位，相比32位系统增加了一倍。
2. 与此对应，64位系统的理论地址空间也从32位系统的 4GB 扩展到了 256TB，同样增加了一倍的数量级。
3. 所以虽然地址空间都有数量级的增加，但地址空间容量与地址总线位宽之间的比例关系仍然保持一致。
4. 比如，在32位系统中比例是 4GB:32bit，在64位系统中该比例则变成了256TB:64bit。
5. 容量级数都成比例地左移了一个数量级，因此两个系统的地址空间与总线位宽的对应关系是一致的。
6. 这种比例关系的一致性，保证了软硬件设计和逻辑兼容性。
综上所述，之所以说64位系统沿用了32位系统的地址布局比例，是因为它们地址空间与位宽之间的对应关系是成比例连贯的，所以可以认为地址布局方式是一致的。这种比例逻辑的继承是64位系统向下兼容的重要基础。

为什么64位系统中的理论最大虚拟地址空间是256TB呢？
因为：
1. 32位系统的地址位宽为32位，所以每个地址可以表示的数量级是2^32，约等于4GB。
2. 而64位系统地址位宽为64位，每个地址可以表示的数量级是2^64，约等于256TB。
3. 从32位扩展到64位，地址位数成双倍增长，所以地址空间的数量级是2的指数级成长，而不是简单的线性增加。
4. 2^64远大于2^32的2倍，所以64位地址空间理论上不是8GB，而是可以达到256TB。
5. 这符合地址空间容量与地址位宽之间的指数对应关系(而非线性对应关系)，地址空间容量 = 2的地址位宽次方。
所以简单来说，地址空间的增长是与地址位数呈指数级对应，而不是线性对称增长，这也是64位系统具有支持极大内存容量的重要原因之一。

大多数需要设置(increaseUserVA)或AWE的问题都可以通过增加的理论最大值来解决。

虽然地址空间分配布局这个概念不能直接转化成Windows内部机制，但是正确理解这一点对于我们利用它来帮助滥用Windows内部机制至关重要。

0x05 动态链接库

Microsoft官方文档将DLL描述为“一个包含代码和数据的库，可以同时被多个程序使用。”

DLL可被用作Windows中应用程序执行背后的核心功能之一。根据Windows文档所述“DLL的使用有助于促进代码模块化、代码重用、高效内存使用和减少磁盘空间，因此，使用了DLL之后，操作系统和程序加载速度更快，运行速度更快，并会占用计算机上更少的磁盘空间 ”。

当DLL作为程序中的函数被加载时，该DLL将被指定为依赖项。由于程序依赖于DLL，因此攻击者可以针对DLL而不是应用程序本身来控制程序执行或程序功能的某些方面。

  • DLL Hijacking-DLL劫持技术 (T1574.001)
  • DLL Side-Loading-DLL侧加载技术(白加黑) (T1574.002)
  • DLL Injection-DLL注入技术 (T1055.001)

DLL的创建方式与创建任何其他项目/应用程序没有什么不同，它们只需要轻微的语法修改即可工作，下面是Visual C++ Win32动态链接库项目中的DLL示例。

#define EXPORTING_DLL
#include "sampleDLL.h"

)
{
    return TRUE;
}

void HelloWorld()
{
    MessageBox( NULL, TEXT("Hello World"), TEXT("In a DLL"), MB_OK);

在创建了DLL之后，我们还要知道如何在应用程序中使用它们。

DLL可以通过使用加载时动态链接(load-time dynamic linking)或者运行时动态链接(run-time dynamic linking)在程序中被加载。

当使用加载时动态链接(load-time dynamic linking)来加载DLL时，会从应用程序中显式调用DLL函数，我们只能通过提供头文件(.h)和导入库文件(.lib)来实现这种类型的链接。下面是一个从应用程序中调用被导出的DLL函数的示例。

#include "stdafx.h"
#include "sampleDLL.h"
int APIENTRY WinMain(HINSTANCE hInstance, HINSTANCE hPrevInstance, LPSTR lpCmdLine, int nCmdShow)
{
    HelloWorld();
    return 0;
}

当使用运行时动态链接(run-time dynamic linking)来加载DLL时，将使用单独的函数(LoadLibrary或者LoadLibraryEx)在运行时加载DLL，完成加载后，还需要使用GetProcAddress来标识将要调用的被导出的DLL函数。下面是一个在应用程序中加载和导入DLL函数的示例。

...

...
HINSTANCE hinstDLL;

BOOL fFreeDLL;

hinstDLL = LoadLibrary("sampleDLL.dll");
if (hinstDLL ! = NULL)
{
    HelloWorld = (DLLPROC) GetProcAddress(hinstDLL, "HelloWorld");
    if (HelloWorld ! = NULL)
        (HelloWorld);
    fFreeDLL = FreeLibrary(hinstDLL);

...

在恶意代码中，恶意攻击者通常会更多地使用运行时动态链接而不是加载时动态链接。这是因为恶意程序可能需要在内存区域之间传输文件，而传输单个DLL比使用其他文件需求(file requirements)进行导入更加易于管理。

我们重新进行筛选，在Procmon中使用过滤器：进程名称为“notepad.exe”，操作为“load image”，路径以“.dll”结尾。

Image Optional Header(可选映像头)有一个具有欺骗性的名称，它实际上是PE文件头(PE file header)的重要组成部分。

Data Dictionaries是image optional header(可选映像头)的一部分，它们包含了多个image data directory结构，每个image data directory结构都用于描述和指向不同的数据目录，这些数据目录可以包括导入表、导出表、资源表等；换句话说，Data Directories中存储了指向实际数据的指针(以image data directory结构形式存在)，这些指针可用于定位和访问文件中的各种数据。

在PE (Portable Executable) 文件格式中，image data directory结构是用于描述和定位不同数据目录的一种数据结构，它包含两个字段：VirtualAddress 和 Size。

  • VirtualAddress（虚拟地址）： 它是一个相对虚拟地址（RVA），指示相应数据目录在文件中的位置。RVA 是一个相对于映像基址的偏移量，用于定位数据目录。
  • Size（大小）： 它表示数据目录中数据的大小，即占用的字节数。

image data directory结构通常用于描述 PE 文件中的关键信息，比如导入表、导出表、资源表等；当PE文件加载到内存时，这些信息可以通过解析相应的数据目录来获取。

总的来说，image data directory结构提供了一种灵活的方式，以允许PE文件存储和定位各种重要的数据结构。

0x07 与Windows内部机制交互

与Windows内部机制进行交互可能看起来令人难以理解，但是该过程已经得到了极大的简化。要与Windows内部机制交互，最容易实现和展开研究的选择是通过Windows API调用来建立接口。Windows API为我们提供了与Windows操作系统进行交互的本地功能(本机功能)，该API包含Win32 API和Win64 API(不太常见)。

在本文内容中，我们仅会提供一些与Windows内部机制相关的特定API调用的简要概述。

大多数Windows内部组件都需要与物理硬件和内存进行交互。

Windows内核将控制所有程序和进程，并架起所有软件和硬件交互的桥梁，这一点尤其重要，因为许多Windows内部结构都需要以某种形式与内存进行交互。

在默认情况下，应用程序通常无法直接与内核交互或者修改物理硬件(因为应用程序需要接口)，这个问题可以通过使用处理器模式和访问级别来解决。

Windows操作系统的处理器具有用户模式和内核模式，处理器将根据实际访问和被请求的模式在user mode和kernel mode之间进行切换。

用户模式和内核模式之间的切换通常是通过系统和API调用来实现，在官方文档中，这个点有时被称为“切换点”。

User mode 用户模式 Kernel Mode 内核模式
没有直接的硬件访问权限 能够直接进行硬件访问
在私有虚拟地址空间中创建进程 在单个共享虚拟地址空间中运行进程
只能访问“该用户已经拥有的内存位置” 可以访问整个物理内存

在用户模式或“用户空间”下启动的应用程序将保持该模式，直到进行系统调用或通过API建立交互接口。当进行系统调用时，应用程序将切换模式，下图是描述此过程的流程图。

当研究编程语言如何与Win32 API交互时，这个过程可能会变得更加复杂，应用程序将在使用API之前先经过编程语言运行时。最常见的例子是C#在与Win32 API进行交互和进行系统调用之前会先经历CLR(Common Language Runtime)执行过程。

tips：CLR(Common Language Runtime)是Microsoft .NET Framework中的核心组件之一，它提供了一种在托管环境中执行和管理.NET应用程序的框架；CLR执行是指通过CLR运行.NET应用程序的过程。

接下来，我们将在本地进程中注入一个消息框(message box)，以演示与内存交互的概念验证(POC：proof-of-concept)。

将消息框写入内存的步骤概述如下：

  1. 为消息框分配本地进程内存。

  2. 将消息框写入/复制到已分配的内存中。

  3. 执行本地进程内存中的消息框。

第一步，我们可以使用OpenProcess来获取指定进程的句柄。

HANDLE hProcess = OpenProcess(
	PROCESS_ALL_ACCESS, // Defines access rights
	FALSE, // Target handle will not be inhereted
	DWORD(atoi(argv[1])) // Local process supplied by command-line arguments 
);

然后，我们可以使用VirtualAllocEx来分配一个带有有效载荷缓冲区(payload buffer)的内存区域。

remoteBuffer = VirtualAllocEx(
	hProcess, // Opened target process
	NULL, 
	sizeof payload, // Region size of memory allocation
	(MEM_RESERVE | MEM_COMMIT), // Reserves and commits pages
	PAGE_EXECUTE_READWRITE // Enables execution and read/write access to the commited pages
);

第二步，我们可以使用WriteProcessMemory将有效载荷(payload)写入已分配的内存区域。

WriteProcessMemory(
	hProcess, // Opened target process
	remoteBuffer, // Allocated memory region
	payload, // Data to write
	sizeof payload, // byte size of data
	NULL
);

第三步，我们可以使用CreateRemoteThread从内存中执行我们所写入的有效载荷(payload)。

remoteThread = CreateRemoteThread(
	hProcess, // Opened target process
	NULL, 
	0, // Default size of the stack
	(LPTHREAD_START_ROUTINE)remoteBuffer, // Pointer to the starting address of the thread
	NULL, 
	0, // Ran immediately after creation
	NULL
);


0x08 总结

Windows内部结构(Windows内部机制)是Windows操作系统工作原理的核心，如果不影响操作系统的基本运行方式，那么这些内部结构就无法被改变；因此，Windows内部结构对于攻击者来说是一个有利可图的攻击目标，攻击者往往会恶意滥用Windows内部组件的功能。

本文内容中所涵盖的许多概念可以被转化为Unix系统中相对应的概念，虽然有关攻击的细节及其执行方式可能会发生变化，但是许多核心概念将保持不变。
