THM-Windows-API介绍

hybcx2024-07-252024-08-01

了解如何与 win32 API 交互并了解其广泛的用例

0x01 介绍

Windows API提供了与Windows操作系统的关键组件进行交互的本地功能。Windows API正在被许多人广泛使用，包括网络安全红队、网络安全威胁参与者、网络安全蓝队、软件开发人员和网络安全解决方案提供商等。

Windows API可以与Windows系统无缝集成，并且能够为我们提供广泛的使用案例。你可能会看到Win32 API被用于攻击性工具和恶意软件开发、EDR(端点检测与响应)工程以及一般的软件应用程序。有关Windows API用例的更多信息，请查看Windows API索引。

本文学习目标

  1. 了解Windows API是什么、Windows API的相关用例有哪些，以及它如何与操作系统子系统进行交互；
  2. 学习如何用不同的编程语言来实现Windows API
  3. 从恶意的角度来理解如何使用Windows API，并且分解实际的API恶意利用案例以供研究；

在开始学习本文相关内容之前，建议读者能够对操作系统架构有一个大致的了解，还建议读者具备基本的编程知识(但这不是必需的)。

本文的目的在于学习有关Windows API的基础知识，我们将简要介绍Win32 API的实现，并且将重点讨论为什么要使用API调用功能以及从何处开始进行API调用。

0x02 与子系统、硬件交互

一个程序通常需要访问或修改Windows子系统或硬件，但是这种访问和修改往往受限于维护机器的稳定性。为了解决这个问题，微软发布了Win32 API，它是一组用于Microsoft Windows操作系统的应用程序编程接口(API)，允许开发者与Windows操作系统进行交互并且访问底层系统功能，这些API包含在Windows操作系统的动态链接库(DLL)中，开发者可以通过在程序中引用这些库来使用这些API。

Windows通过两种不同的模式来区分硬件访问情况：用户模式和内核模式。这些模式决定了应用程序或驱动程序被允许访问的硬件、内核和内存，而API或系统调用会在这两种模式之间提供接口，从而将信息发送到系统以便在内核模式下进行处理。

User mode 用户模式 Kernel mode 内核模式
没有直接的硬件访问权限 能够直接进行硬件访问
只能访问“该用户已经拥有的内存位置” 可以访问整个物理内存

有关内存管理的详细信息，请参考博文：【THM】Windows Internals(Windows内部机制基础)-红队

在研究编程语言如何与Win32 API交互时，上述过程可能会变得更加复杂，应用程序将在使用API之前先经过编程语言运行。最常见的例子是C#在与Win32 API进行交互和进行系统调用之前会先经历CLR(Common Language Runtime)执行过程。

当你以管理员身份启动一个应用程序时，实际上你是以更高的用户权限（例如root用户或Windows中的管理员用户）来运行该程序。这意味着该程序在用户模式下运行，但拥有更高的用户权限，可以执行一些普通用户无法执行的操作，例如：

- 修改系统配置文件。
- 访问受限的系统资源。
- 安装或卸载软件。
- 执行其他需要管理员权限的操作。

然而，这并不意味着该程序会运行在内核模式下。即使以管理员身份启动，应用程序仍然运行在用户模式下，只是具有更高的用户权限。

### 什么时候运行在内核模式下

只有操作系统的核心组件和一些特殊的系统服务，如设备驱动程序和内核模块，才会在内核模式下运行。这些组件在设计上需要直接访问硬件资源和系统内存，因此需要在内核模式下运行。

### 总结

以管理员身份启动应用程序并不会在内核模式下打开进程。该程序仍然在用户模式下运行，但拥有更高的用户权限。内核模式运行通常仅限于操作系统的核心组件和设备驱动程序。

0x03 Windows API的组件

Win32 API(通常被称为Windows API)具有多个依赖组件，这些组件可用于定义API的结构和组织形式。

让我们通过自上而下的方式拆解Win32 API，我们将假设API是顶层，而组成特定调用的参数是底层；在下面的表格中，我们将概括地描述这种自上而下的结构，并在后文中进行详细介绍。

Layer 层 Explanation 解释
API 用于描述Win32 API结构中的任何调用的顶层/通用术语。
头文件或导入项 定义要在运行时被导入的库(由头文件或库导入定义)，使用指针来获取相关函数地址
核心DLLs(核心动态链接库) 一组用于定义调用结构的四个 DLL(KERNEL32、USER32、GDI32.DLL和ADVAPI32)，这些 DLL 定义了不包含在单一子系统中的内核和用户服务
附加 DLLs(补充性动态链接库) 被定义为Windows API的一部分的其他DLLs，这些DLLs控制着Windows操作系统的独立子系统，大约有36个已定义的其他DLLs(包括NTDLL、COM、FVEAPI等)
调用结构(或调用方式) 定义API调用本身以及调用的参数。
API调用 在程序中使用的API调用，从指针中获取相关函数地址
输入/输出参数 由调用结构定义的参数值

在下文内容中，我们将继续讨论导入库、核心头文件和调用结构等概念，我们将更加深入地研究“调用”，并且了解在何处(以及如何)进行调用以便更好地理解调用参数、变量。

可以在维基百科中，搜索windows.h，依靠关键词进行进一步搜索

0x04 操作系统库(OS Libraries)

Win32库的每个API调用都驻留在内存中，并且需要一个指向内存地址的指针，而由于ASLR(Address Space Layout Randomization-地址空间布局随机化)的实施，获取这些函数指针的过程将变得模糊，为此每种编程语言或软件包都有独特的过程来克服ASLR；在本文中，我们将讨论两种最流行的用于克服ASLR以便在API调用过程中获取函数指针的实现：Platform Invoke和Windows头文件。

Windows头文件

微软已经发布了Windows头文件，通过使用Windows API和头文件可以为我们提供关于ASLR实现问题的间接解决方案，从概念层面上来理解：在运行时(runtime)，加载程序将确定正在进行哪些函数调用，并创建一个thunk table(用于处理不同调用约定的中间层结构)以获取函数地址或指针。
相关官方文档：

https://docs.microsoft.com/en-us/windows/win32/winprog/using-the-windows-headers
或
https://learn.microsoft.com/zh-cn/windows/win32/winprog/using-the-windows-headers

一旦windows.h文件被包含在非托管程序的顶部，就允许实现任何Win32函数的调用。

P/Invoke

微软将P/Invoke或平台调用描述为“一种允许你从托管代码中访问非托管库中的结构、回调和函数的技术”。

相关官方文档：

https://learn.microsoft.com/en-us/dotnet/standard/native-interop/pinvoke

或

https://learn.microsoft.com/zh-cn/dotnet/standard/native-interop/pinvoke

P/invoke提供了一些工具，用于处理从托管代码调用非托管函数(或者换句话说，调用 Win32 API)的整个过程。P/invoke将通过导入包含非托管函数或Win32 API调用的所需DLL来启动，下面是导入带有选项的DLL的示例。

public class Program
{
[DllImport("user32.dll", CharSet = CharSet.Unicode, SetLastError = true)]
...
}

总结

Platform Invocation Services（P/Invoke）是.NET Framework提供的一项服务，用于在托管代码中调用非托管代码，特别是调用Windows API或其他本地库的函数，这项服务允许在托管环境中声明和调用非托管函数，使得.NET应用程序可以与底层平台进行交互。

P/Invoke允许.NET代码调用C、C++、汇编等非托管语言编写的函数，并在参数、返回值等方面进行交互，这种能力对于与操作系统或其他原生库进行集成以执行底层任务非常重要。

使用P/Invoke的主要步骤包括：

  1. 声明非托管函数： 在托管代码中使用DllImport特性声明非托管函数。这个特性提供了关于如何调用该函数的信息，包括函数名、库的名称和其他调用约定等。
[DllImport("user32.dll", CharSet = CharSet.Unicode, SetLastError = true)]
private static extern int MessageBox(IntPtr hWnd, string lpText, string lpCaption, uint uType);
  1. 调用非托管函数： 在.NET代码中像调用任何其他托管函数一样调用已声明的非托管函数。
MessageBox(IntPtr.Zero, "Hello, P/Invoke!", "Greetings", 0);

//在这个调用示例中，IntPtr.Zero 用于表示空窗口句柄（hWnd 参数），字符串 "Hello, P/Invoke!" 作为消息文本（lpText 参数），"Greetings" 作为消息框标题（lpCaption 参数），整数值 0 则作为 uType 参数。

通过P/Invoke，.NET开发者可以借助非托管代码的功能，访问系统级别的功能，执行底层操作，以及与其他编程语言或原生库进行集成。

拓展

在.NET框架中，“托管”（Managed）和"非托管"（Unmanaged）是两个关键的概念，它们描述了在.NET应用程序中运行的代码的性质。

  1. 托管代码（Managed Code）：

    • 托管代码是在.NET运行时环境下执行的代码。
    • 它由.NET编程语言（如C#、VB.NET）编写，并在运行时由.NET运行时（Common Language Runtime，CLR）进行管理。
    • 托管代码受到CLR的管理和控制，包括内存管理、垃圾回收、类型安全性等。
    • 托管代码在运行时受到.NET框架的约束和监控，具有更高的安全性和可靠性。

  2. 非托管代码（Unmanaged Code）：

    • 非托管代码是在不受.NET运行时管理的环境中执行的代码。
    • 它可以是使用其他编程语言（如C、C++）编写的、直接针对操作系统或硬件的代码。
    • 非托管代码通常需要开发者手动管理内存、资源，而不依赖于CLR的垃圾回收机制。
    • 与托管代码相比，非托管代码可能更容易导致内存泄漏、安全漏洞等问题。

在.NET应用程序中，通常会同时存在托管和非托管代码，例如通过使用P/Invoke（Platform Invocation Services），开发者可以在托管代码中调用非托管的原生函数。这种托管代码和非托管代码混合使用的能力使得.NET平台可以与底层的原生库、操作系统进行交互，同时也充分利用了CLR所提供的高级特性。

0x05 API调用结构

API调用是Win32库的主要组件之一，这些调用提供了可扩展性和灵活性，能够用于满足大量用例。大多数Win32 API调用都有比较完整的文档记录，我们可以通过访问以下链接来了解更多相关信息：Windows API documentation以及pinvoke.net。

在本小节中，我们将简单介绍API调用的命名方案和输入/输出参数。

API调用功能可以通过修改命名方案和附加表示性字符来扩展，下面是Microsoft所支持的关于命名方案的字符表。

Character 字符 Explanation 解释
A 表示采用ANSI编码的8位字符集
W 表示 Unicode 编码
Ex 为 API 调用提供扩展功能或输入/输出参数

有关此概念的详细信息，请查看相关的Microsoft文档。

每个API调用还有一个预定义的结构来定义它的输入/输出参数，你可以在Windows文档的相应API调用文档页面中找到其中的大多数结构，以及关于每个I/O参数的说明。

我们以WriteProcessMemory API调用为例，以下是从相关文档页面所获取到的关于此调用的 I/O结构：

BOOL WriteProcessMemory(
  [in]  HANDLE  hProcess,
  [in]  LPVOID  lpBaseAddress,
  [in]  LPCVOID lpBuffer,
  [in]  SIZE_T  nSize,
  [out] SIZE_T  *lpNumberOfBytesWritten
);

对于每个I/O参数，Microsoft文档还解释了它的用途、预期的输入或输出以及可接受的值，即使有了官方文档的解释，确定这些值有时候对于特定的调用也是具有挑战性的事情，因此我们建议在具体代码中使用调用之前，最好先尝试研究并找到相关的API调用的使用示例。

0x06 C API 实现

微软为C和C++等低级编程语言提供了一组预配置的库，我们可以使用这些库来访问所需的API调用。

如前所述(第四小节)，windows.h头文件可被用于定义调用结构并获取函数指针。如果要包含windows头文件，请将以下代码行添加到任何C或C++程序之前即可。

#include <windows.h>

让我们直接开始创建第一个 API 调用，作为我们的第一个示例，我们的目标是使用CreateWindowExA来创建一个标题为“Hello THM!”的弹出窗口，为了重申上文第五小节中所涉及的内容，让我们先观察一下相关调用的in/out参数。

HWND CreateWindowExA(
  [in]           DWORD     dwExStyle, // Optional windows styles
  [in, optional] LPCSTR    lpClassName, // Windows class
  [in, optional] LPCSTR    lpWindowName, // Windows text
  [in]           DWORD     dwStyle, // Windows style
  [in]           int       X, // X position
  [in]           int       Y, // Y position
  [in]           int       nWidth, // Width size
  [in]           int       nHeight, // Height size
  [in, optional] HWND      hWndParent, // Parent windows
  [in, optional] HMENU     hMenu, // Menu
  [in, optional] HINSTANCE hInstance, // Instance handle
  [in, optional] LPVOID    lpParam // Additional application data
);

让我们使用上述预定义的参数并为它们赋值，正如第五小节中所述，关于API调用中的每个参数都有对其用途和潜在值的说明。下面是对CreateWindowsExA的完整调用的示例。

HWND hwnd = CreateWindowsEx(
	0,
	CLASS_NAME,
	L"Hello THM!",

我们已经用c语言定义了第一个API调用，现在我们可以将其实现到应用程序中，并使用API调用的功能。下面是一个示例应用程序，该程序将使用API来创建一个小的空白窗口。

BOOL Create(
        PCWSTR lpWindowName,
        DWORD dwStyle,
        DWORD dwExStyle = 0,
        int x = CW_USEDEFAULT,
        int y = CW_USEDEFAULT,
        int nWidth = CW_USEDEFAULT,
        int nHeight = CW_USEDEFAULT,
        HWND hWndParent = 0,
        HMENU hMenu = 0
        )
    {
        WNDCLASS wc = {0};

        wc.lpfnWndProc   = DERIVED_TYPE::WindowProc;
        wc.hInstance     = GetModuleHandle(NULL);
        wc.lpszClassName = ClassName();

        RegisterClass(&wc);

        m_hwnd = CreateWindowEx(
            dwExStyle, ClassName(), lpWindowName, dwStyle, x, y,
            nWidth, nHeight, hWndParent, hMenu, GetModuleHandle(NULL), this
            );

        return (m_hwnd ? TRUE : FALSE);
    }

如果一切顺利，当我们运行上述示例应用程序之时，我们将看到一个标题为“Hello THM!”的窗口成功弹出。

正如本小节所演示的案例一样，使用低级编程语言使得快速定义API调用变得非常容易，而由于C语言的易于使用性和可扩展性，基于C的编程语言在网络安全威胁参与者和网络安全供应商中很受欢迎。

0x07 .NET和PowerShell API实现

如第四小节中所述，P/Invoke允许我们导入DLLs并将指针分配给API调用，为了理解P/Invoke是如何实现这种效果的，让我们先看一下以下示例：

class Win32 {
	[DllImport("kernel32.dll")]
	public static extern IntPtr GetComputerNameA(StringBuilder lpBuffer, ref uint lpnSize);
}

这段代码是一个使用 P/Invoke（Platform Invocation Services）的 C# 类，用于调用 Windows API 中的 GetComputerNameA 函数，该函数的目的是获取计算机的名称。

让我们逐行解析这个代码段：

  1. class Win32: 这是一个名为 Win32 的 C# 类。

  2. [DllImport("kernel32.dll")]: 这是一个特性，用于指示编译器去寻找名为 kernel32.dll 的库，并在其中查找 GetComputerNameA 函数。DllImport 是 P/Invoke 的一部分，它允许 C# 代码调用非托管 DLL 中的函数。

  3. public static extern IntPtr GetComputerNameA(StringBuilder lpBuffer, ref uint lpnSize): 这是一个用于声明对 GetComputerNameA 函数的引用的 C# 方法。它包括以下几个部分：

    • public: 这表示方法是公开的，可以从类外部访问。
    • static: 这表示方法是静态的，可以通过类名直接调用，而不需要创建类的实例。
    • extern: 这表示方法是在外部实现的，也就是说，它是通过 P/Invoke 调用非托管代码实现的。
    • IntPtr: 这是返回类型，表示函数返回一个指针。在这里，GetComputerNameA 函数返回计算机名的字符串的指针。
    • GetComputerNameA: 这是方法的名称，对应于 kernel32.dll 中的函数名称。
    • (StringBuilder lpBuffer, ref uint lpnSize): 这是方法的参数列表。StringBuilder 用于接收计算机名的字符串，ref uint 用于接收字符串的长度。

这个方法的作用是调用 Windows API 中的 GetComputerNameA 函数，将计算机名存储在传递的 StringBuilder 中，并通过 ref uint 参数返回计算机名的长度。

上述的class Win32类函数通常将存储已定义的API调用以及要在所有将来的方法中引用的定义。

接下来我们必须使用DllImport特性来导入存储了API调用结构的库，导入的DLLs的作用类似于头文件包，但是要求基于我们所寻找的API调用来导入指定的DLL；我们可以参考API索引或pinvoke.net来确定特定的API调用在DLL中的位置。

在进行DLL导入时，我们可以创建一个指向我们想要使用的API调用的新指针(在C#中是由 IntPtr 类型所定义的指针)，与一些低级语言不同，在.NET中导入DLL时我们还必须在指针中指定in/out参数结构，正如上文第五小节中所提及的那样，我们可以在相关的Windows文档中找到我们所需的API调用的in/out参数。

现在我们可以在应用程序中实现上述已定义的API调用并使用其功能，下面是一个示例应用程序，它将使用API来获取运行它的设备的计算机名称以及其他信息。

class Win32 {
	[DllImport("kernel32.dll")]
	public static extern IntPtr GetComputerNameA(StringBuilder lpBuffer, ref uint lpnSize);
}

static void Main(string[] args) {
	bool success;
	StringBuilder name = new StringBuilder(260); // 260 是缓冲区的初始大小
	uint size = 260;
	success = GetComputerNameA(name, ref size);
	Console.WriteLine(name.ToString());
}
//需要注意的是，这个例子中的缓冲区(name)大小是一个初始值，如果获取的计算机名比缓冲区大，GetComputerNameA函数会修改 `lpnSize` 参数并返回所需的缓冲区大小，你需要根据返回的大小重新调整缓冲区(name)的大小，然后再次调用函数以确保缓冲区(name)有足够的空间来存储计算机名。

如果一切顺利，那么上述示例程序的运行结果应该返回当前设备的计算机名称。

现在，我们已经介绍了如何在.NET中完成API实现，接下来让我们看看如何调整相同的语法以便在PowerShell中完成类似示例。

在PowerShell中定义API调用几乎与在.NET中相同，但是我们需要创建一个方法而不是一个类，并且还要添加一些额外的运算符。

$MethodDefinition = @"

#   - 这是一个 C# 方法的定义，使用了 [DllImport] 特性，它声明了一个叫做 GetProcAddress 的函数，该函数用于获取一个模块中导出函数的地址。

0x08 常见的被滥用的API调用

Win32库中的多个API调用很容易被恶意活动利用。

一些网络实体会试图记录和组织所有针对API调用的可用恶意载体，如： SANs、MalAPI.io。

虽然有很多API调用会被滥用，但是有些API调用的在恶意利用会更加常见，下表列出了在一组样本中最经常被滥用的API(按频率排列)：

API Call (API 调用) Explanation (解释)
LoadLibraryA 将指定的DLL映射到调用进程的地址空间中
GetUserNameA 检索与当前线程关联的用户的名称
GetComputerNameA 检索本地计算机的NetBIOS或DNS名称
GetVersionExA 获取关于当前所运行的操作系统的版本信息
GetModuleFileNameA 检索指定模块和进程的文件的完全限定路径
GetStartupInfoA 检索STARTUPINFO结构的内容(窗口站、桌面、标准句柄和进程的外观)
GetModuleHandle 返回指定模块的模块句柄 (前提是指定模块已经映射到调用进程的地址空间中)
GetProcAddress 返回指定被导出的DLL函数的地址
VirtualProtect 更改对调用进程的虚拟地址空间中的内存区域的保护

tips：上表内容中的"映射" 指的是将模块加载到进程的内存中，使得它可以被进程访问和执行。

Keylogger(键盘记录器)

为了开始分析键盘记录器，我们需要收集它正在实现哪些API调用和钩子(hooks)。由于作为本小节示例的键盘记录器是用C#编写的，因此它必须使用P/Invoke来获取每个调用的指针，下面是键盘记录器示例中关于p/invoke定义的源代码片段：

[DllImport("user32.dll", CharSet = CharSet.Auto, SetLastError = true)]
private static extern IntPtr SetWindowsHookEx(int idHook, LowLevelKeyboardProc lpfn, IntPtr hMod, uint dwThreadId);
[DllImport("user32.dll", CharSet = CharSet.Auto, SetLastError = true)]
[return: MarshalAs(UnmanagedType.Bool)]

[DllImport("kernel32.dll", CharSet = CharSet.Auto, SetLastError = true)]
private static extern IntPtr GetModuleHandle(string lpModuleName);
private static int WHKEYBOARDLL = 13;
[DllImport("kernel32.dll", CharSet = CharSet.Auto, SetLastError = true)]

//1. `SetWindowsHookEx` 函数声明：
//    ```csharp

//    ```
//   - 该声明用于调用 `SetWindowsHookEx` 函数，该函数用于设置(安装)钩子。
//   - `idHook` 是钩子类型的标识符，这里使用了 `WHKEYBOARDLL`，表示低级键盘输入事件的钩子。
//   - `lpfn` 钩子过程的委托，通常是一个回调函数，用于处理接收到的钩子消息。

//    private static extern bool UnhookWindowsHookEx(IntPtr hhk);

//   - 该声明用于调用 `GetModuleHandle` 函数，该函数用于获取指定模块的句柄。
//  - `lpModuleName` 是要获取句柄的模块的名称，我们可以使用ProcessName属性来获取当前进程的进程名以指定模块，也可以直接将lpModuleName设置为 null 或者 IntPtr.Zero，这样GetModuleHandle函数将返回调用进程的模块句柄。

tips：在计算机编程和软件开发领域，“hooks” 通常指的是一种机制或技术，它用于截取或拦截系统、应用程序中特定事件的发生；通过使用hooks，开发者可以在事件发生前或事件发生后注入自定义代码，从而改变或扩展系统、应用程序的行为。

tips：“钩子过程”（hook procedure）是在安装了系统或应用程序的钩子后，用于处理特定事件的由用户定义的函数；钩子是一种机制，允许应用程序监视、拦截或修改操作系统中的消息、事件或者其他形式的处理流程。

以下是关于每个API调用及其各自用法的说明：

API Call (API 调用) Explanation (解释)
SetWindowsHookEx 将内存钩子安装到钩子链中以监视某些事件
UnhookWindowsHookEx 从钩子链中移除已安装的钩子
GetModuleHandle 返回指定模块的模块句柄 (前提是指定模块已经映射到调用进程的地址空间中)
GetCurrentProcess 检索当前进程的伪句柄(pseudo handle)

tips：在 Windows 操作系统中，有两种类型的句柄：实际句柄和伪句柄。

  1. 实际句柄： 实际句柄是操作系统内部分配的实际资源句柄，用于标识和操作系统资源交互。例如，文件句柄、线程句柄、进程句柄等都属于实际句柄。
  2. 伪句柄： 伪句柄是一个在逻辑上标识资源但不直接与实际系统资源关联的值。伪句柄是为了方便编程而引入的。在很多情况下，Windows API 使用伪句柄来表示当前进程或当前线程，这些伪句柄无需对底层资源进行实际操作，只是一种标识符。

上表中的GetCurrentProcess 所返回的是表示当前进程的伪句柄，伪句柄不是典型的内核对象句柄，这个伪句柄不直接映射到实际的系统资源，而是作为标识符，方便进行与当前进程相关的操作。

使用伪句柄而不是实际句柄的一个好处是：不需要关闭或释放伪句柄，因为它们不引用实际的资源。因此，GetCurrentProcess 返回的伪句柄只是用于标识当前进程并用于引用当前进程的一种方便的方式。

接下来，我们将分析键盘记录器示例如何在当前进程上设置钩子，下面是键盘记录器示例中的钩子部分的源代码片段。

public static void Main() {
	_hookID = SetHook(_proc);
	Application.Run();
	UnhookWindowsHookEx(_hookID);
	Application.Exit();
}
private static IntPtr SetHook(LowLevelKeyboardProc proc) {
	using (Process curProcess = Process.GetCurrentProcess()) {
		return SetWindowsHookEx(WHKEYBOARDLL, proc, GetModuleHandle(curProcess.ProcessName), 0);
	}
}
//这是一个使用 Windows 钩子（Windows Hooks）监控低级键盘事件的 C# 代码片段。让我解释一下主要部分：
//Main 方法是应用程序的入口点。
//_hookID 是一个变量，用于存储设置的钩子的标识符。
//SetHook 方法用于设置键盘钩子。LowLevelKeyboardProc 是一个代表低级键盘过程的委托（delegate）。

//具体的解释如下：
//Main 方法：
//_hookID = SetHook(_proc); 调用 SetHook 方法设置键盘钩子，并将返回的标识符存储在 _hookID 中。
//Application.Run(); 启动应用程序的消息循环，使应用程序保持运行状态。
//UnhookWindowsHookEx(_hookID); 在应用程序退出前解除键盘钩子。
//Application.Exit(); 退出应用程序。

//SetHook 方法：
//using (Process curProcess = Process.GetCurrentProcess()) 获取当前进程的 Process 对象。
//SetWindowsHookEx(WHKEYBOARDLL, proc, GetModuleHandle(curProcess.ProcessName), 0) 调用 SetWindowsHookEx 函数来设置键盘低级钩子。
//WHKEYBOARDLL 是用于指定键盘低级钩子的常量。
//proc 是用于处理低级键盘事件的回调函数。
//GetModuleHandle(curProcess.ProcessName) 获取当前进程的模块句柄。
//0 表示将钩子安装在所有线程上。

//注意，这段代码片段在使用钩子的同时涉及到系统级的操作，如键盘事件的监控。使用钩子是一个强大而敏感的操作，需要小心处理以避免潜在的问题，此外，根据使用场景和权限，可能需要以管理员身份运行应用程序。

让我们了解键盘记录器的使用目标和实现步骤，然后从示例代码片段中确定相关的API调用情况。

Shellcode启动器

要开始分析shellcode启动器，我们需要收集它正在实现哪些API调用，这个过程看起来应该与上一个案例研究相同。下面是shellcode启动器示例中关于p/invoke定义的一个源代码片段：

private static UInt32 MEM_COMMIT = 0x1000;
private static UInt32 PAGE_EXECUTE_READWRITE = 0x40;
[DllImport("kernel32.dll")]
private static extern UInt32 VirtualAlloc(UInt32 lpStartAddr, UInt32 size, UInt32 flAllocationType, UInt32 flProtect);

private static extern IntPtr CreateThread(UInt32 lpThreadAttributes, UInt32 dwStackSize, UInt32 lpStartAddress, IntPtr param, UInt32 dwCreationFlags, ref UInt32 lpThreadId);

//1. VirtualAlloc 函数声明：
//   ```csharp
//   private static UInt32 MEM_COMMIT = 0x1000;//此处的0x1000表示将内存块提交为可用的物理存储器，即将一块虚拟地址空间映射到实际的物理存储器上。
//   private static UInt32 PAGE_EXECUTE_READWRITE = 0x40; //此处的0x40表示内存页具有可执行、可读、可写的保护属性。

//   [DllImport("kernel32.dll")]
//   private static extern UInt32 VirtualAlloc(UInt32 lpStartAddr, UInt32 size, UInt32 flAllocationType, UInt32 flProtect);
//   ```
//   - `VirtualAlloc` 函数用于在进程的虚拟地址空间中分配内存。
//   - `MEM_COMMIT` 是用于指定分配内存的选项之一。
