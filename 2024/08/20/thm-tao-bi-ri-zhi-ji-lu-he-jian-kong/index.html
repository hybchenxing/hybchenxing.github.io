THM-逃避日志记录和监控

hybcx2024-08-202024-08-20

了解如何使用与工具无关的现代方法绕过常见的日志记录和系统监控，例如 ETW。

0x01 介绍

攻击者路径中最大的障碍之一是日志记录和监控。与防病毒和EDR （端点****检测和响应）解决方案不同，日志记录创建可以分析恶意活动的物理活动记录。

如何监控设备将取决于公司的环境和偏好。团队可能决定根本不监控某些设备。通常，监控解决方案将从主机设备开始，收集应用程序或事件日志。创建日志后，它们可以保存在设备上或发送到事件收集器/转发器。一旦它们离开设备，防御团队就会决定如何聚合它们；这通常是使用索引器和SIEM （安全信息和事件管理器）来完成的。

一旦从设备上获取日志，攻击者可能没有太多控制权，但可以控制设备上的内容以及获取日志的方式。攻击者的主要目标是事件日志，由ETW ( E vent T racing for Windows ) 管理和控制。

该房间将解决事件跟踪及其弱点，以允许攻击者逃避或禁用基于 ETW 的解决方案。

学习目标

  • 了解事件追踪的技术和实现。
  • 了解如何创建技术来逃避 ETW。
  • 了解如何将理论规避概念应用于代码。

在开始本课程之前，请先熟悉 Windows 的基本用法和功能；我们建议完成Windows 内部房间。还建议您具备 C 和PowerShell的基本编程知识，但不是必需的。

我们提供了一台基本的 Windows 机器以及完成这个房间所需的文件。您可以使用以下凭据在浏览器中或通过RDP访问计算机。

Machine IP: MACHINE_IP Username: Administrator Password: Tryhackme!

这将包含大量信息。请系好安全带并找到最近的灭火器。

0x02 事件追踪

如前所述，Windows 中的几乎所有事件日志记录功能都是由 ETW 在应用程序和内核级别处理的。虽然还有事件日志记录和*跟踪日志记录等其他服务，*但这些服务要么是 ETW 的扩展，要么对攻击者来说不太常见。

Component Purpose

Providers 供应商 Generate events 生成事件
Consumers 消费者 Interpret events 解读事件

我们将在下一个任务中更深入地介绍每个组件以及如何对其进行检测。

虽然对于攻击者来说，事件 ID 的重要性不如组件，但它是 Windows 日志记录的核心功能。事件以XML （可扩展标记语言）格式发送和传输，这是提供商如何定义和实现事件的标准。以下是事件 ID 4624 的示例：帐户已成功登录。

至此，我们明白了为什么日志记录可以扰乱攻击者，但是 ETW 与攻击者到底有何关系呢？ ETW 对大多数操作系统具有可见性，而日志记录通常具有有限的可见性或细节。

由于 ETW 的可见性，攻击者应始终注意执行操作时可能生成的事件。消除 ETW 的最佳方法是在保持环境完整性的同时，尽可能限制其对您正在做的事情的洞察力。

在接下来的任务中，我们将涵盖 ETW 检测、ETW 规避和其他基于 ETW 的解决方案。

0x03 日志规避方法

在深入研究更现代和技术性的规避技术之前，让我们先了解一下可用的各种方法及其对攻击者和防御者的影响。

当第一次考虑和评估日志规避时，您可能会认为简单地销毁或篡改日志可能是可行的。

遵循安全最佳实践，现代环境通常采用日志转发。日志转发意味着SOC将日志从主机移动或“转发”到中央服务器或索引器。即使攻击者可以从主机中删除日志，它们也可能已经脱离设备并受到保护。

假设攻击者在转发日志之前确实销毁了所有日志，或者如果没有转发日志，这将如何引发警报？攻击者首先必须考虑环境完整性；如果没有来自设备的日志，则可能会引起严重怀疑并导致调查。即使攻击者确实控制了删除和转发的日志，防御者仍然可以跟踪篡改行为。

Event ID Purpose
1102 清除 Windows 安全审核日志时记录
104 清除日志文件时记录
1100 Windows事件日志服务关闭时记录

上述事件ID可以监控销毁日志或“日志粉碎”的过程。这给试图篡改或破坏日志的攻击者带来了明显的风险。尽管可以进一步绕过这些缓解措施或篡改日志，但攻击者必须评估风险。在接近某个环境时，您通常不了解安全实践，并且尝试此方法会面临OPSEC （操作安全）风险。

tips：操作安全 (OPSEC) 是一组用于保护操作员或操作安全的原则和策略。例如，使用代号而不是真实姓名，或者使用代理来隐藏您的 IP 地址。

如果之前的做法过于激进，我们该如何从战略上解决这个问题呢？

攻击者必须关注恶意技术可能导致的日志，以保持环境的完整性。了解可能针对他们的工具后，他们可以利用或修改已发布的方法。

大多数已发布的技术都将针对 ETW 组件，因为这将允许攻击者最大程度地控制跟踪过程。

这个房间将分解一些最常见的已发表技术和允许广泛控制的更现代的技术。

0x04 追踪仪器

ETW 分为三个独立的组件，共同管理和关联数据。 Windows 中的事件日志与通用XML数据没有什么不同，因此易于处理和解释。

事件控制器用于构建和配置会话。为了扩展此定义，我们可以将控制器视为确定数据如何流动和流向何处的应用程序。根据 Microsoft 文档，“控制器是定义日志文件的大小和位置、启动和停止事件跟踪会话、启用提供程序以便它们可以将事件记录到会话、管理缓冲池的大小以及获取会话的执行统计信息的应用程序。”

事件提供者用于生成事件。为了扩展这个定义，控制器将告诉提供者如何操作，然后从其指定源收集日志。根据Microsoft 文档，“提供程序是包含事件跟踪工具的应用程序。提供程序注册自身后，控制器可以启用或禁用提供程序中的事件跟踪。提供者定义其对启用或禁用的解释。一般来说，启用的提供程序会生成事件，而禁用的提供程序则不会。”

还有四种不同类型的提供商，支持各种功能和遗留系统。

Provider Purpose
MOF （托管对象格式） 定义 MOF 类的事件。一次由一个跟踪会话启用。
WPP （ Windows软件跟踪预处理器） 与TMF （ Trace Message Format ）文件关联以解码信息。一次由一个跟踪会话启用。
Manifest-Based 基于清单的 定义清单中的事件。一次最多可启用八个跟踪会话。
TraceLogging 跟踪记录 包含所有必需信息的自描述事件。一次最多可启用八个跟踪会话。

事件消费者用于解释事件。为了扩展这个定义，消费者将选择会话并同时解析该会话或多个会话中的事件。这在“*事件查看器”中最常见。*根据Microsoft 文档，“使用者是选择一个或多个事件跟踪会话作为事件源的应用程序。消费者可以同时从多个事件跟踪会话请求事件；系统按时间顺序传递事件。消费者可以接收存储在日志文件中的事件，或者从实时传送事件的会话中接收事件。”

从开始到结束，事件均源自提供者。控制器将确定数据发送到何处以及如何通过会话进行处理。消费者将保存或传送日志以供解释或分析。

现在我们了解了 ETW 是如何检测的，这对攻击者有何影响？我们之前提到过在保持完整性的同时限制可见性的目标。我们可以通过定位组件来限制洞察力的特定方面，同时维护大部分数据流。下面是针对每个 ETW 组件的特定技术的简短列表。

Component Techniques
Provider 提供者 PSEtwLogProvider 修改、组策略接管、日志管道滥用、类型创建
Controller 控制器 修补 EtwEventWrite、运行时跟踪篡改、
Consumers 消费者 日志粉碎、日志篡改

我们将在接下来的任务中深入介绍这些技术，以提供一个巨大的可能性工具箱。

0x05 欢乐与静谧的反射

在PowerShell中，ETW 提供程序从**.NET 程序集**PSEtwLogProvider加载到会话中。根据Microsoft 文档，“程序集构成了基于 .NET 的应用程序的部署、版本控制、重用、激活范围和安全权限的基本单元。” .NET 程序集可能看起来很陌生；然而，我们可以通过知道它们以熟悉的格式形成，例如 exe（可执行文件）或 dll（动态链接库）来使它们更加熟悉。

在 PowerShell 会话中，大多数 .NET 程序集在启动时加载到与用户相同的安全上下文中。由于会话与加载的程序集具有相同的权限级别，因此我们可以通过PowerShell反射修改程序集字段和值。 O’Reilly表示，“反射允许您查看程序集内部并找出其特征。在 .NET 程序集中，存储了描述程序集包含内容的信息。这称为元数据。从某种意义上说，.NET 程序集是，自我描述，至少如果询问正确的话。”

在ETW （ Windows的Event Tracing）上下文中，攻击者可以反射 ETW 事件提供程序集并将字段m_enabled设置为$null 。

从较高层次来看， PowerShell反射可以分为四个步骤：

  1. 获取PSEtwLogProvider的 .NET 程序集。
  2. 为etwProvider字段存储空值。
  3. 将m_enabled字段设置为之前存储的值。

第一步，我们需要获取PSEtwLogProvider程序集的类型。存储程序集是为了在下一步中访问其内部字段。

$logProvider = [Ref].Assembly.GetType('System.Management.Automation.Tracing.PSEtwLogProvider')

在第二步中，我们存储要使用的前一个程序集中的值 ( $null )。

$etwProvider = $logProvider.GetField('etwProvider','NonPublic,Static').GetValue($null)

在第三步中，我们将步骤一起编译，以使用前一行中存储的值覆盖m_enabled字段。

[System.Diagnostics.Eventing.EventProvider].GetField('m_enabled','NonPublic,Instance').SetValue($etwProvider,0);

我们可以将这些步骤一起编译并将它们附加到恶意PowerShell脚本中。使用提供的PowerShell脚本并尝试此技术。

为了证明该脚本的有效性，我们可以执行它并测量给定命令返回的事件数。

0x06 针对 ETW 特定技术

修补跟踪函数

ETW 从每个新进程的运行时加载，通常源自CLR （通用语言运行时）。在新进程中，ETW 事件从用户区发送并直接从当前进程发出。攻击者可以将预定义的操作码写入 ETW 的内存中函数来修补和禁用功能。在深入研究该技术的具体细节之前，让我们先观察一下修补在高层次上可能是什么样子。从最基本的定义来看，我们试图强制应用程序在到达我们想要修补的功能之前退出或返回。

为了更好地理解这个概念，我们创建了一个基本的伪函数，它将执行数学运算，然后返回一个整数。如果在原始回车之前插入回车，则程序将无法完成后续行。

int x = 1
int y = 3
return x + y

// output: 4  
    
int x = 1
return  x
int y = 3
return x + y

// output: 1

使这个高级概念适应我们的目标，如果我们能够确定如何在内存中调用返回值，我们就可以将其写入函数并期望它在任何其他行之前运行。我们期望返回值放在顶部**，因为堆栈使用LIFO** （后进先出）结构。上面是 LIFO 结构如何工作的简要图表。当我们更深入地研究这项任务时，我们将扩展后进先出结构的运作方式。

现在我们对 return 语句和 LIFO 结构有了更多的了解，让我们回到它如何应用于事件跟踪。在编写任何代码或确定修补函数的步骤之前，我们需要确定恶意函数以及我们可以从中返回的可能点。通过之前的研究，我们知道在 CLR 中，ETW 是通过函数EtwEventWrite编写的。为了识别“补丁点”或返回，我们可以查看函数的反汇编。

779f2459 33cc		       xor	ecx, esp
779f245b e8501a0100	   call	ntdll!_security_check_cookie
779f2460 8be5		       mov	esp, ebp
779f2462 5d		         pop	ebp
779f2463 c21400		     ret	14h

在观察函数时，我们正在寻找将返回函数或停止函数执行的操作码。通过研究或者熟悉汇编指令，我们可以确定ret 14h将结束函数并返回到之前的应用程序。

根据IA-32 文档，“ ret指令将控制转移到位于堆栈上的返回地址。”

用更专业的术语来说，ret 将弹出堆栈中最后一个值。 ret ( 14h ) 的参数将指定出栈后释放的字节数或字数。

为了中和该函数，攻击者可以将ret14h 、 c21400的操作码字节写入内存来修补该函数。

为了更好地理解我们试图在堆栈上实现的目标，我们可以将操作码应用于之前的 LIFO 图。

现在我们已经对该技术背后的核心原理有了基本的了解，让我们看看它的技术应用方式。

在较高层面上，ETW 修补可以分为五个步骤：

  1. 获取EtwEventWrite的句柄
  2. 修改函数的内存权限
  3. 将操作码字节写入内存
  4. 重置该功能的内存权限（可选）
  5. 刷新指令缓存（可选）

第一步，我们需要获取EtwEventWrite地址的句柄。该函数存储在ntdll中。我们将首先使用LoadLibrary加载库，然后使用GetProcAddress获取句柄。

var ntdll = Win32.LoadLibrary("ntdll.dll");
var etwFunction = Win32.GetProcAddress(ntdll, "EtwEventWrite");

第二步，我们需要修改函数的内存权限，以允许我们写入该函数。函数的权限由flNewProtect参数定义； 0x40启用 X、R 或 RW 访问（内存保护约束）。

uint oldProtect;
Win32.VirtualProtect(
	etwFunction, 
	(UIntPtr)patch.Length, 
	0x40, 
	out oldProtect
);

在第三步，该函数具有我们需要写入它的权限，并且我们有预定义的操作码来修补它。因为我们正在写入一个函数而不是一个进程，所以我们可以使用臭名昭著的Marshal.Copy来编写我们的操作码。

patch(new byte[] { 0xc2, 0x14, 0x00 });
Marshal.Copy(
	patch, 
	0, 
	etwEventSend, 
	patch.Length
);

在第四步，我们可以开始清理步骤以将内存权限恢复原样。

VirtualProtect(etwFunction, 4, oldProtect, &oldOldProtect);

在第五步，我们可以确保修补后的函数将从指令缓存中执行。

Win32.FlushInstructionCache(
	etwFunction,
	NULL
);

我们可以将这些步骤编译在一起并将它们附加到恶意脚本或会话中。使用提供的 C# 脚本并尝试此技术。

操作码写入内存后，我们可以再次查看反汇编函数来观察补丁。

779f23c0 c21400		    ret	14h
779f23c3 00ec		      add	ah, ch
779f23c5 83e4f8		    and	esp, 0FFFFFFF8h
779f23c8 81ece0000000	sub	esp, 0E0h

在上面的反汇编中，我们准确地看到了 LIFO 图（图 2 ）中所描绘的内容。

一旦函数在内存中被修补，当调用EtwEventWrite时它将始终返回。

尽管这是一种精心设计的技术，但根据您的环境，它可能不是最好的方法，因为它可能限制比您想要的完整性更多的日志。

经由策略的提供者

ETW 具有大量开箱即用的覆盖范围，但除非指定，否则它将禁用某些功能，因为它们可以创建大量日志。可以通过修改其父策略的GPO （组策略对象**）设置来启用这些功能。两个最流行的 GPO 提供商提供了PowerShell的覆盖范围，包括脚本块日志记录和**模块日志记录。

脚本块日志记录将记录在PowerShell会话中执行的任何脚本块。 ETW 提供程序在PowerShell v4 中引入并在PowerShell v5 中改进，它有两个要报告的事件 ID。

Event ID Purpose
4103 Logs command invocation 记录命令调用
4104 Logs script block execution 记录脚本块的执行

事件 ID 4104 对于攻击者来说最为常见，如果未正确混淆或隐藏，可能会暴露他们的脚本。下面是 4104 日志的简短示例。

Event ID:4104
Source:Microsoft-Windows-PowerShell
Category:Execute a Remote Command
Log:Microsoft-Windows-PowerShell/Operational
Message:Creating Scriptblock text (1 of 1):
Write-Host PowerShellV5ScriptBlockLogging

模块日志记录是一个非常详细的提供程序，它将记录从其发送的任何模块和数据。在PowerShell v3 中引入， PowerShell会话中的每个模块都充当提供者并记录其自己的模块。与之前的提供程序类似，模块会将事件写入事件 ID 4103。下面是 4103 日志的示例。

由于创建的日志量较大，事件 ID 4103 对于攻击者来说不太常见。这通常会导致其得到较轻的治疗或完全禁用。

尽管攻击者有可用的 ETW 补丁，但它们可能并不总是实用或逃避日志记录的最佳方法。作为替代方案，攻击者可以针对这些提供商来缓慢限制可见性，同时不像其他技术那样明显或嘈杂。

禁用这些提供程序的总体目标是限制您所需组件的可见性，同时仍然使环境看起来未被篡改。

组策略接管

模块日志记录和脚本块日志记录提供程序均通过组策略启用，特别是Administrative Templates -> Windows Components -> Windows PowerShell 。如任务 4 中所述，在PowerShell会话中，系统程序集与用户加载到相同的安全上下文中。这意味着攻击者与缓存 GPO 设置的程序集具有相同的权限级别。使用反射，攻击者可以获取实用程序字典并修改任一PowerShell提供程序的组策略。

在高层，组策略接管可以分为三个步骤：

  1. 从实用程序缓存中获取组策略设置。
  2. 将通用提供程序修改为0 。
  3. 修改调用或模块定义。

我们将分解一个示例PowerShell脚本来识别每个步骤，并在下面深入解释每个步骤。

第一步，我们必须使用反射来获取System.Management.Automation.Utils的类型并识别GPO缓存字段： cachedGroupPolicySettings 。

$GroupPolicySettingsField = [ref].Assembly.GetType('System.Management.Automation.Utils').GetField('cachedGroupPolicySettings', 'NonPublic,Static')
$GroupPolicySettings = $GroupPolicySettingsField.GetValue($null)

在第二步中，我们可以利用GPO变量将任一事件提供程序设置修改为0 。 EnableScriptBlockLogging将控制4104事件，限制脚本执行的可见性。可以通过直接写入对象或注册表来完成修改。

$GroupPolicySettings['ScriptBlockLogging']['EnableScriptBlockLogging'] = 0

在第三步中，我们可以使用任何其他提供程序设置重复上一步，我们希望EnableScriptBlockInvocationLogging将控制4103事件，从而限制 cmdlet 和管道执行的可见性。

$GroupPolicySettings['ScriptBlockLogging']['EnableScriptBlockInvocationLogging'] = 0

我们可以将这些步骤一起编译并将它们附加到恶意PowerShell脚本中。使用提供的PowerShell脚本并尝试此技术。

注意：该脚本的核心功能与上述代码相同，但略有修改以符合PowerShell v.5.1 更新。

滥用日志管道

在 PowerShell 中，每个模块或管理单元都有一个任何人都可以用来修改其日志记录功能的设置。根据Microsoft 文档，“当LogPipelineExecutionDetails属性值为 TRUE ( $true ) 时，Windows PowerShell会将会话中的 cmdlet 和函数执行事件写入事件查看器中的 Windows PowerShell日志。”攻击者可以在任何PowerShell会话中将此值更改为$false以禁用该特定会话的模块日志记录。 Microsoft 文档甚至指出了从用户会话禁用日志记录的功能，“要禁用日志记录，请使用相同的命令序列将属性值设置为 FALSE ( $false )。”

在高层，日志管道技术可以分为四个步骤：

  1. 获取目标模块。
  2. 将模块执行详细信息设置为$false 。
  3. 获取模块管理单元。
  4. 将管理单元执行详细信息设置为$false 。
$module = Get-Module Microsoft.PowerShell.Utility # Get target module
$module.LogPipelineExecutionDetails = $false # Set module execution details to false
$snap = Get-PSSnapin Microsoft.PowerShell.Core # Get target ps-snapin
$snap.LogPipelineExecutionDetails = $false # Set ps-snapin execution details to false

上面的脚本块可以附加到任何PowerShell脚本或在会话中运行以禁用当前导入模块的模块日志记录。

0x07 实践

在这种情况下，您是一名红队操作员，被分配构建一个规避脚本来禁用 ETW 并执行已编译的二进制文件。在这种情况下，环境完整性至关重要，蓝队正在积极监控环境。您的团队已告知您，他们主要关注的是监控网络流量；如果停止，它们可能会提醒您的连接。蓝队还被认为正在搜索可疑日志；但是，他们不转发日志。使用在这个房间中获得的知识，创建一个脚本来不受干扰地执行二进制文件或命令。

$logProvider = [Ref].Assembly.GetType('System.Management.Automation.Tracing.PSEtwLogProvider')
$etwProvider = $logProvider.GetField('etwProvider','NonPublic,Static').GetValue($null)
[System.Diagnostics.Eventing.EventProvider].GetField('m_enabled','NonPublic,Instance').SetValue($etwProvider,0);

$module = Get-Module Microsoft.PowerShell.Utility # Get target module
$module.LogPipelineExecutionDetails = $false # Set module execution details to false
$snap = Get-PSSnapin Microsoft.PowerShell.Core # Get target ps-snapin
$snap.LogPipelineExecutionDetails = $false # Set ps-snapin execution details to false

正如在整个房间中提到的，逃避事件检测的主要目标是尽可能保持环境干净和完整，同时防止记录会话或代码。
